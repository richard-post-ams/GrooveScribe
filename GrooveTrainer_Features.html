Feature Roadmap

Planned features beyond the POC scope, organised by development phase.

Requirements v1.3 Last updated: June 2026 Owner: Richard
POC nice-to-have Ship in POC if time allows
Groove detail / tip panel High

An expandable description below the groove title giving the student context before they start playing. Especially important for Rudiments where the exercise purpose is not obvious from the name alone.

  • Short description shown by default (1-2 lines, always visible)
  • Expandable "Why practice this?" section with pedagogical context
  • Tip field: optional technique note authored by admin (e.g. "Keep wrists relaxed")
  • Collapsed by default, expands on tap, remembers state per session
  • Admin can author both fields in the groove editor
Affects: groove player screen, groove editor (admin)
Admin: drag-to-reorder grooves High

In the Groove Manager, admin can reorder grooves within a level and category by dragging. The sort order determines the sequence students see and the "next groove" flow after completion.

  • Drag handles visible on each groove row in the admin list
  • Reordering updates the sortOrder field immediately in localStorage
  • Student groove list and "next groove" CTA reflect the new order without page reload
  • Up/down arrow buttons as fallback for touch devices where drag is unreliable
Affects: admin groove manager
Reset progress on a single groove Medium

A student can clear all tempo checkpoints for a specific groove and start fresh. Useful when returning after a break or wanting to re-practice from the beginning.

  • "Reset progress" link below the tempo chip row, low-prominence styling
  • Confirmation dialog: "Reset all progress for this groove?" with Confirm / Cancel
  • On confirm: all chips return to grey, progress saved to localStorage
  • Does not affect progress on any other groove
  • Groove list card updates to "Not started" state immediately after reset
Affects: groove player screen, groove list
Visual feedback on tempo chip completion Medium

A small animation when a student marks a tempo done, making the progress moment feel rewarding. CSS-only, no libraries.

  • Chip bounces once (scale 1 → 1.2 → 1) on mark-done tap
  • Chip turns green with a brief fill transition (200ms)
  • On final chip: entire chip row pulses green once, completion banner slides in
  • On mark-undone: chip returns to grey, no animation needed
  • Animations respect prefers-reduced-motion: disabled if system setting is on
Affects: groove player screen only
Post-POC After POC is validated
Audio error handling High

Mobile browsers block audio autoplay. The app needs graceful handling when MIDI fails to load or play, particularly on first load on iOS Safari.

  • On audio load failure: show "Tap to enable audio" prompt before first play
  • iOS Safari requires a user gesture to unlock AudioContext — handle this explicitly
  • If MIDI engine fails to initialise: show clear error with retry button, not a silent broken state
  • Network offline: show "You're offline — playback unavailable" banner, progress marking still works
Affects: groove player, app init
Offline mode Medium

If the Google OAuth session token expires and there is no network, the app should not show a broken login screen. Students should still be able to practice using the cached session.

  • On app load offline: check localStorage for cached Google user ID
  • If cached session exists: bypass auth check, load app in read-only mode
  • Progress marking still works (localStorage)
  • Show subtle "Offline mode" indicator in header
  • On reconnect: re-validate session silently in background
Affects: auth layer, app shell
Audio settings Low

Volume controls for the two distinct audio channels: drum sounds and metronome click. Some students want to hear only the metronome while they play along on a real kit.

  • Two sliders in a settings panel: "Drums" volume and "Metronome" volume
  • Sliders range 0–100, default both at 80
  • Settings persist in localStorage per user
  • Muting drums (0) shows "Play along mode" badge in the player
  • Accessible from a ⚙ icon in the groove player header
Affects: groove player, settings layer
Future Requires backend or subscription infrastructure
Cross-device progress sync High

Progress currently lives in localStorage and does not sync across devices. A backend (Firebase or Supabase free tier) would unlock this and is a prerequisite for subscription.

  • Replace localStorage progress store with Firestore or Supabase table
  • Progress syncs on login, on every checkpoint mark, and on app focus
  • Conflict resolution: last-write-wins per individual BPM checkpoint
  • Offline writes queued and synced on reconnect
Requires: backend, auth upgrade
Subscription and paywall High

Monetisation layer. White Stick free for all users, Yellow–Red Stick behind subscription. Stripe integration for payment.

  • Stripe Checkout for subscription signup
  • Free tier: White Stick fully accessible
  • Paid tier: all stick levels, early access to new content
  • Locked level cards show "Subscribe to unlock" CTA
  • Admin dashboard shows subscriber count (read-only from Stripe)
Requires: backend, Stripe integration
Teacher-student assignment Low

Admin (teacher) can assign specific grooves or levels to individual students and see their progress. Transforms the platform from self-study to guided learning.

  • Admin can view a student list and their per-groove progress
  • Admin can assign a "practice plan" — a curated list of grooves for a student
  • Student sees assigned grooves highlighted on their home screen
  • Progress notifications: admin gets notified when a student completes an assigned groove
Requires: backend, email/push notifications
Video lesson embed Low

Each groove or rudiment can have an optional YouTube or Vimeo embed showing a teacher demonstrating the exercise. Admin adds the video URL in the groove editor.

  • Video thumbnail shown in groove detail, expands inline on tap
  • No autoplay — student initiates playback explicitly
  • Video pauses automatically when the drum player starts (avoid audio clash)
Requires: no backend, but needs groove editor update
